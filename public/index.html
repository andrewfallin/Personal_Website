IOS Secret Santa App

(Swift)
At a glance
  • Secret Santa gift exchange manager app
  • Swift and IOS Software Development Kit
  • Allows gift exchange planner to participate without prior assignment knowledge

In Depth

Secret Santa gift exchange manager app. The app allows users to create multiple Secret Santa Gift exchanges. When creating their exchange, the users are able to add as many participants as they would like by entering their name and email. These are then added to a table where the user may delete a participant if they would like before beginning the exchange. They also set a price cap and the date of the exchange. Once the user clicks start, the exchange will begin and all participants (including the user) receive an email with the exchange information (price cap, rules, date) as well as who they are assigned to give a gift to.

On another page the user is able to manage current exchanges. If a participant A were to leave the exchange then an email would be sent to the participant B, who was giving a gift to participant A, informing them their giftee had left the exchange and reassign them to the giftee of participant A.

Additionally, if the user were to delete the exchange entirely then all participants would receive an email informing them the exchange had been canceled.

I created this when I realized that the person who was planning the gift exchange missed out on the best part of Secret Santa, not knowing who is getting you a gift. The app takes care of all assignments so everyone can participate. To accomplish this, I programmed in Swift and used the IOS Software Development Kit.

Voice Control Module for CASAS research group

(Python)
At a glance
  • Voice Control Module for controling smart home's robotic assistant
  • Software: Python, XMPP, Redis, Open Source Jasper Project, Festival (TTS), Wit.ai (STT)
  • Hardware: Raspberry Pi 3B+, Blinkt LED strip, USB speaker, Conference Microphone
  • Took in user commands, parsed them into json object with additional information (location, sound level), then sent command to CASAS Middleware
  • Project Leader

In Depth

The Center for Advanced Studies in Adaptive Systems (CASAS) is a research group dedicated to the research and design of smart homes. More specifically, CASAS’ goal is to install smart tech in elderly patient’s homes to provide them with more autonomy while not sacrificing the assisted care they need. If a patient were to injure themselves or have a medical issue, then the house would contact the proper authorities.

One aspect the the research group is beginning to implement is a Robot Activity Support System (RAS). The RAS is being designed to help patients with general reminders for daily tasks. The need for a more intuitive interface with the robot was quickly identified. Therefore, CASAS tasked my senior design team with building a voice control system that could be installed in the home and provide patients with a simpler mode of interaction.

After talking with the team, we decided that we wanted to try to keep the voice control module “in-house” as much as possible, without relying on outside services such as AWS. Because of this we chose to build our system on a Raspberry Pi 3B+. The CASAS team also stated that they wanted two forms of system feedback, so we added a Blinkt LED strip to the GPIO pins and a USB speaker so the system could provide audio feedback. Further, our system took in voice commands through a conference microphone.

For the software, we built off of an open source voice control framework called Jasper. Jasper provided a very intuitive way for adding new commands. You simply had to add a new file to the modules folder, tell it the keywords to look out for, and then run the program. On runtime, all files in the modules folder are loaded as commands.

When the user gives a command to the system, the command is saved into a NoSQL database Redis. In addition to the command, Redis also stores the location (XYZ coordinates and room name) of each Raspberry Pi. Another process is running called Jasper Connect. This process checks to see if the Redis key exists. If it does then the value stored at that key, as well as the location and the measured sound level, are compiled into a JSON object and published to the CASAS XMPP middleware. The robot team then is able to pull the information down to their systems and manipulate it as necessary.

As project leader, my responsibilities included scheduling meetings with our team and clients, assigning tasks to the team, and submitting all generated documents. I was also in charge of writing code to connect and send commands to the middleware.

File System for the MTX Operating System

( C )
At a glance
  • Built working file system
  • Pointer and String manipulation
  • Followed basic linux commands for system interaction

In Depth

Our file system was built in a tree-like manner using C. If the command ‘mkdir’ was entered, for example, a new directory with a pointer back to a directory is created. Additionally, the new directory is given the permissions to store files. If the command ‘touch’ is entered, then a new file is created in the current directory. A new pointer is created that points from the current directory to the file, however; the file is not given permission to store additional directories.

Other commands that we supported in our file system were ‘cd’, ‘ls’, ‘pwd’, ‘link’, ‘read’, ‘rmdir’, ‘creat’, ‘write’, ‘cat’, ‘cp’, ‘open’, and ‘close’. Each of these involved different levels of string and pointer manipulation. For example, ‘cd’ would take the current working directory pointer and reassign it to point it at the user specified directory.

Bullet Hell Shooter Game

( C# )
At a glance
  • Object Oriented Programming techniques such as Factory and Decorator patterns
  • Team of 6 students
  • MonoGame

In Depth

Built a Bullet Hell style shooter game with 5 other students using Object Oriented Programming techniques. Our game featured a player and a series of levels the player would work through, including two boss enemies. All of our enemies inherited from one base enemy class, their differences being defined by how much health they have. Each of these enemies was built using the factory pattern in an effort to add a single point of origin for each of our enemies. This is also how we created all of our bullets.

For each of enemies we used decorator to define their abilities. For example, our enemies’ movements and extra health were added as a “decoration” when each of the enemies was created.
